# AI Native PPT 编辑器
一个面向学术场景的、AI与用户协同的演示文稿编辑器。核心设计理念是：将AI定位为“可被调度的协作者”，而非不可干预的主导者，让用户在叙事结构层面持续调整意图，内容与结构保持联动。

设计方案，包含四个部分。

## 1)行为观察与用户思考
- 目标用户：研究生、科研人员、学术机构从业者，日常需要解读文献、做组会汇报、写综述报告。
真实行为观察（基于用户自述及身边同伴访谈）：
用户使用现有AI PPT工具（如doubao、秘塔ai）生成学术汇报PPT，全程约40分钟。

- 常见痛点：
页面模板化严重：每页都是“标题+要点+图片”的固定结构，缺乏对论文逻辑（如“问题-方法-结果-讨论”的层次）的深度理解。
无法干预生成过程：AI一次生成全部内容，用户只能被动接受，然后导出到PowerPoint手动逐页修改——这实际上是“用AI生成初稿，再用传统工具二次加工”，效率提升有限。
内容与结构割裂：用户想调整某一部分的侧重（比如想多讲“方法”少讲“背景”），但AI不理解这种“叙事意图”的变更，只能重新生成全文。
缺乏并行与对比：对于长论文，AI按顺序逐页生成，用户无法同时看到多个章节的草稿，也难以及时叫停某个不满意的部分。
- __核心洞察：
学术PPT的本质是结构化叙事，而非简单的要点罗列。用户需要的是“可编辑的结构蓝图”，而不是一次性成品。
用户希望AI扮演多线程协作者：能同时生成不同章节，允许用户随时“打断”并校正某一模块，其他模块继续工作。
真正的AI原生工作流应该是非线性、可干预、结构优先的。__

## 2)产品设计思路
产品结构：AI Native PPT 编辑器采用“三层结构 + 核心亮点”模型。

```
┌─────────────────────────────────────────────┐
│                全局叙事蓝图                   │
│  （可编辑的大纲树形结构，支持拖拽重组）         │
└─────────────────────────────────────────────┘
          │ 联动          │ 联动
┌─────────▼─────────┐  ┌──▼──────────────────┐
│   多智能体工坊      │  │   实时内容预览区     │
│ - 并行生成卡片      │  │ - 所见即所得编辑     │
│ - 独立暂停/校正     │  │ - 直接修改内容       │
│ - 任务进度可视化    │  │ - 触发结构同步       │
└───────────────────┘  └─────────────────────┘
```
#### __核心亮点：__
__叙事蓝图优先__
>- 用户上传论文后，AI首先生成的是可编辑的结构化大纲（而非直接生成页面）。用户可以像操作思维导图一样，增删章节、调整顺序、修改层级。这个蓝图成为后续所有AI生成任务的总控。
  
__可打断的并行智能体__
>- 将每个章节的生成任务分配给独立的AI智能体（可理解为不同Prompt的模型实例）。所有任务在工坊面板中同时进行，每个任务都显示进度条和当前生成的片段。

>- 用户可以暂停某个章节的生成，修改该章节的生成指令（例如：“方法部分不要写得过细，重点突出创新点”），然后恢复生成。
被修改的指令会实时反馈到该智能体，后续内容基于新指令重新生成，而其他未暂停的章节不受影响。
__这解决了“用户无法插手”的核心痛点。__

__内容-结构双向联动__
>- 用户在内容预览区直接编辑某页的文字或配图，系统会自动判断是否需要更新叙事蓝图（例如新增了一个小节），并询问是否同步调整大纲。
反之，用户在蓝图中调整结构时，内容区也会自动重排，并调用AI重新生成受影响的部分（只做增量修改，而非全量）。

__总览检查机制__
>- 所有章节生成完毕后，AI会启动一个“审查智能体”，对整个PPT进行一致性检查（如术语统一、逻辑连贯性、引用完整性）。检查结果以批注形式列出，点击任意批注可直接跳转到对应的章节工坊卡片，允许用户快速修正。



## 3)用户完整动线
> 场景：研究生小张需要将一篇10页的机器学习论文制作为组会汇报PPT。

- 动线描述（重点对比传统AI PPT工具）：


| 步骤 | StructuSlide中的行为 | 与传统AI PPT的差异 |
|------|----------------------|---------------------|
| **1. 上传论文** | 拖拽PDF/上传，系统自动解析全文。 | 相同起点，但传统工具直接进入“模板选择”。 |
| **2. 选择解读侧重点** | 系统提供4种预设Prompt模式：「综述概览」「方法深挖」「实验复现」「批判性讨论」。小张选择「方法深挖」。 | 传统工具通常只生成通用学术模板，不区分侧重点。 |
| **3. 生成叙事蓝图** | AI返回树形大纲，包含“背景→相关工作→核心方法（细分为3个子节点）→实验设计→结果分析→结论”。小张手动将“实验设计”拖拽到“核心方法”之后，并删除“相关工作”。 | 传统工具直接生成固定页序，用户要改只能导出后手动调顺序。 |
| **4. 启动并行生成** | 蓝图确认后，系统将每个章节拆分为独立任务，工坊面板同时显示6个任务卡片，每个卡片显示生成进度和实时生成的草稿文本。 | 传统工具顺序生成全部页面，用户等待5-10分钟，期间无法干预。 |
| **5. 中途校正** | 小张发现“核心方法”部分生成的算法描述过于简略，他**暂停**该任务，修改生成指令为：“请详细写出算法伪代码步骤，并配一个流程图”，然后**恢复**任务。其他章节继续生成不受影响。 | 无法干预，只能等全部完成后整体修改。 |
| **6. 内容微调与联动** | 在内容预览区，小张直接编辑了“实验设计”页中的表格数据。系统弹出提示：“检测到内容变更，是否同步更新叙事蓝图中的‘关键指标’节点？”小张选择“是”，蓝图自动补充了一个子节点。 | 内容与结构完全割裂，调整内容不会影响大纲。 |
| **7. 总览检查** | 全部任务完成后，点击“智能审查”，系统返回3条建议：①“术语‘准确率’和‘精确率’混用”②“第4页缺少引用来源”③“结论部分与摘要重复”。小张点击第一条，直接跳转到对应章节卡片并修改。 | 无此功能，用户需自行逐页检查。 |
| **8. 导出与分享** | 导出为PPTX或PDF，支持保留编辑历史。 | 同样可导出，但StructuSlide的最终文件已经过多次迭代，导出后基本无需再改。 |
